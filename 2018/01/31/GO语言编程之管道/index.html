GO语言编程之管道

今天我们来讲讲go语言编程中的管道（通道）概念。

1.知识要点：

  • 管道的概念
  • 管道的运行机制与作用
  • 用go语言实现管道的程序编写与线程的结合应用

（一）管道的概念

在go语言编程中，会有一个管道（或者称为通道）的这么一个概念和知识。那么何为管道（channel）？ 引用别人的解释就是：管道是Go语言在语言级别上提供的goroutine（线程）间的通讯方式，我们可以使用channel在多个goroutine之间传递消息。channel是进程内的通讯方式，是不支持跨进程通信的，如果需要进程间通讯的话，可以使用Socket等网络方式。 说白了管道是用来实现一个进程内的线程与线程之间传递信息（数据）的。 管道有这么几个特点：

  • 一个管道只能传递一种类型的值
  • 管道中的数据是先进先出的

(二)管道的运行机制与作用

我们用代码来解释

看如下代码1：

package main

import "fmt"

func main() {
	// 创建一个缓冲管道，缓冲容量为1
	c := make(chan int, 1)
/*创建一个非缓冲管道，程序是这样写：
c:=make(chan int)
或者这样写：
var c chan int
*/
	// 查看写入管道中数据的数量
	fmt.Println(len(c))
	// 查看管道的缓冲容量
	fmt.Println(cap(c))
	c <- 10 //写入数据，这条管道只能写入int类型,
	fmt.Println(len(c))
	fmt.Println(cap(c))
  fmt.Printf("管道c的类型是：%T\n", c)
	fmt.Println(len(c))
	fmt.Println(cap(c))
}

代码1在终端输出的结果：

$ go run 001_管道.go 0 1 1 1 管道c的类型是：chan int 1 1 ***

  • 第一个输出打印出0，是因为还没往缓冲管道写入数据，所以写入数量为0
  • 对于缓冲管道，我们可以发现它写入数据后，不一定要运行读取的程序后，其他程序才能接着执行，换言之，它把数据写入后，可以缓存在管道里，然后计算机可以去执行“c <- 10”语句下面其他语句命令

goroutine 1 [chan receive]: main.main() /Users/mymac/Downloads/学习/区块链学习/0130课堂案例/001_管道.go:22 +0x27b exit status 2 *** 代码4

package main

import "fmt"

func main() {
	// 创建一个缓冲管道，缓冲容量为1
	//c := make(chan int, 1)
/*创建一个非缓冲管道，程序是这样写：*/
  c:=make(chan int)
/*或者非缓冲管道这样写：
var c chan int
*/
	// 查看写入管道中数据的数量
	fmt.Println(len(c))
	// 查看管道的缓冲容量
	fmt.Println(cap(c))
	c <- 10 //写入数据，这条管道只能写入int类型,
	fmt.Println(len(c))
	fmt.Println(cap(c))
	//<-c //从管道里面取值
	//fmt.Println("从管道里面取值：", <-c)
	a := <-c //将管道里面取出来的值赋给a
	fmt.Println("a = ", a)
	//c <- 20
	//k, ok := <-c //从管理里面取值赋给k，如果赋值成功，
	//ok为true
	//fmt.Println(k, "  ", ok)
	fmt.Println(len(c))
	fmt.Println(cap(c))

}

代码4运行结果如下：

$ go run 001_管道.go 0 0 fatal error: all goroutines are asleep - deadlock!

goroutine 1 [chan send]: main.main() /Users/mymac/Downloads/学习/区块链学习/0130课堂案例/001_管道.go:13 +0x172 exit status 2 ***

  • 代码2出现报错的情况，deadlock表明出现死锁。 原因是非缓冲管道不同于缓冲管道，它缓冲容量为0，也就是它执行写入数据的语句命令后必须执行有读取数据的语句命令，之后其他语句命令才能被执行，不然的话就会出现程序停滞阻塞的情况，即死锁。
  • 代码3没有写入数据的命令，却有读取管道数据的语句，同样出现死锁，整个程序在执行“a := <-c”语句时，陷入停滞阻塞，说明它在等待读取数据（可是没有写入数据的命令，所以等待不到数据来读取），这是非缓冲管道的运行机制的结果。
  • 代码4虽然增加了读取管道内数据的语句命令，但是由于写入数据的语句和读取数据的语句在同一线程内（都在主线程内），语句命令从上到下执行下来在“c <- 10”语句时就让主线程阻塞在这里，根本执行不到“a := <-c”这条读取管道数据的命令，导致死锁。
  • 解决这种死锁问题的方法是把非缓冲管道改成缓冲管道，或者创建分线程，并把写入管道数据的语句命令或者读取管道数据的语句命令由分线程来执行。
  • 代码5和代码6是修改后的正确程序。

  • 代码5是改成缓冲管道后程序运行正常输出打印的结果
  • 还需要提出的一点就是，管道内数据被读取出来后，管道的写入数量会相对应的减少，所以倒数第二个输出打印为0

代码6

package main

import "fmt"

func main() {
	// 创建一个管道
	//c := make(chan int, 1)
	c := make(chan int)
	// 查看写入管道中数据的数量
	fmt.Println(len(c))
	// 查看管道的缓冲容量
	fmt.Println(cap(c))
	go func() {
		c <- 10 //写入数据，这条管道只能写入int类型
	}()
	fmt.Println(len(c))
	fmt.Println(cap(c))
	//管道c的类型是chan int
	//fmt.Printf("管道c的类型是：%T\n", c)
	// <-c //从管道里面取值
	// fmt.Println("从管道里面取值：", <-c)
	a := <-c //将管道里面取出来的值赋给a
	fmt.Println("a = ", a)
	//c <- 20
	//k, ok := <-c //从管理里面取值赋给k，如果赋值成功，
	//ok为true

	//fmt.Println(k, "  ", ok)
	fmt.Println(len(c))
	fmt.Println(cap(c))

}

代码6运行结果：

$ go run 001_管道.go 0 0 0 0 a = 10 0 0 ***

  • 代码6是把写入管道数据的语句放到分线程里，让写入数据的语句和读取管道数据的语句分别在两个线程里，并发执行，不管是写入还是读取，都会相互等待。

package main

import "fmt"

func main() {
	// 创建一个管道
	//c := make(chan int, 1)
	c := make(chan int)
	// 查看写入管道中数据的数量
	fmt.Println(len(c))
	// 查看管道的缓冲容量
	fmt.Println(cap(c))
	//go func() {
  c <- 10 //写入数据，这条管道只能写入int类型
	//}()
	fmt.Println(len(c))
	fmt.Println(cap(c))
	//管道c的类型是chan int
	//fmt.Printf("管道c的类型是：%T\n", c)
	// <-c //从管道里面取值
	// fmt.Println("从管道里面取值：", <-c)
	go func(){
    a := <-c //将管道里面取出来的值赋给a
  }()
	fmt.Println("a = ", a)
	//c <- 20
	//k, ok := <-c //从管理里面取值赋给k，如果赋值成功，
	//ok为true

	//fmt.Println(k, "  ", ok)

}

代码7运行结果：

$ go run 001_管道.go 0 0 fatal error: all goroutines are asleep - deadlock!

  • 代码7中把读取管道数据的语句放在了一个分线程里，但还是出现报错的情况，显示deadlock表明还是死锁。究其原因还是因为主程序一开始语句从上年到下执行，先执行到语句“c <- 10”就让程序陷入停滞阻塞，根本运行不到创建分线程的那段语句命令（也就是go关键字创建的分线程语句）分线程还没有被创建，所以也就轮不到读取管道数据的语句执行，从而出现死锁。
